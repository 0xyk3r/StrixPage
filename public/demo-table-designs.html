Frosted Glass Table — 毛玻璃数据表格
完全自定义实现的表格组件，使用 Nebula 设计系统的毛玻璃效果。表头使用等宽字体（Geist Mono）以增强技术感。 行悬停有微妙的背景变化。搜索栏和表格统一在圆角容器中。此方案完全替换 NaiveUI DataTable， 提供最大的样式控制权，但需要自行实现排序、展开行、树形数据等高级功能。

DESIGN_CONCEPT::B
Fluid Cards — 流体卡片列表
每行数据以独立卡片形式呈现，圆角边框配合悬停微位移效果。适合数据密度较低的场景。 视觉上更现代、更有呼吸感。但对于大量数据和高密度场景（如日志查询、文件列表）， 卡片间距会占用更多垂直空间。此方案同样完全替换 NaiveUI DataTable。

DESIGN_CONCEPT::C
Hybrid — NaiveUI DataTable + Nebula 样式覆盖
推荐方案。保留 NaiveUI DataTable 的核心功能（展开行、树形数据、懒加载、远程分页、选择列）， 同时通过 NaiveUI ThemeOverrides + 全局 CSS 深度覆盖来统一 Nebula 视觉风格。 此方案的改动量最小、风险最低，且能保留已有的 23 个数据表页面的全部业务逻辑不变。 仅需修改：搜索栏区域、操作按钮渲染函数、标签渲染函数、和全局表格样式覆盖。

实现策略

1. NaiveUI ThemeOverrides — 在 App.vue 中为 DataTable 组件设置圆角、背景色、边框色、hover 色等

2. 全局 SCSS 覆盖 — 创建 nebula-table.scss，使用 ::v-deep / 全局选择器覆盖 NaiveUI 内部 DOM 节点的间距、字体、过渡等

3. 操作列重构 — 重写 handleOperate() 工具函数，将 NButton 替换为自定义 Nebula 样式按钮

4. StrixTag 升级 — 更新 StrixTag 组件使用 Nebula pill 样式标签替代 NTag

5. 分页组件 — 可考虑自定义分页样式覆盖 或 保留 NaiveUI NPagination (已通过 ThemeOverrides 适配)

ANALYSIS::COMPARISON
三种方案对比分析
综合考虑样式控制力、开发成本、功能完整性和维护性的全面对比。

A: Frosted Glass (全自定义)

✓ 样式 100% 可控
✓ 与 Nebula 设计无缝融合
✓ 体积更小（无 NaiveUI 表格代码）
✗ 需要自行实现展开行
✗ 需要自行实现树形数据
✗ 需要自行实现懒加载
✗ 需要自行实现选择列
✗ 改动量极大（23个页面）
✗ 测试回归风险高

B: Fluid Cards (全自定义)

✓ 视觉最具特色
✓ 现代卡片化设计
△ 低密度数据效果最佳
✗ 高密度数据浪费垂直空间
✗ 展开行、树形数据实现复杂
✗ 需要完全重写 23 个页面
✗ 不适合日志/文件等列表
✗ 维护成本高
✗ 测试回归风险最高

C: Hybrid（推荐）

✓ 保留全部高级功能
✓ 展开行/树形/懒加载无需重写
✓ 改动量最小、风险最低
✓ 23 个页面业务逻辑不变
✓ 通过 ThemeOverrides 适配
✓ 可逐步增强定制化程度
△ 样式控制约 85%
△ 部分内部结构难以覆盖
✓ 可与 A 方案组合使用

推荐策略：C + A 混合方案

核心思路：使用方案 C（NaiveUI DataTable + Nebula 样式覆盖）作为基础，同时将搜索栏、操作按钮、标签等"周边组件"替换为完全自定义的 Nebula 组件（方案 A 的元素）。

具体做法：

① DataTable 主体 → 保留 NaiveUI，通过 ThemeOverrides + SCSS 覆盖实现 Nebula 风格

② 搜索栏 → StrixBlock 已经是自定义组件（已完成 Nebula 化），内部输入框保留 NaiveUI（通过 ThemeOverrides 已适配）

③ 操作按钮 → 重写 handleOperate()，使用 Nebula 风格图标按钮替代 NButton

④ 标签/状态 → 升级 StrixTag，使用 Nebula pill 样式替代 NTag

⑤ 分页 → 保留 NaiveUI NPagination（可通过 ThemeOverrides 调整风格），或后续替换为自定义分页

⑥ 弹窗/表单 → 保留 NaiveUI NModal + NForm（通过 ThemeOverrides 适配圆角、颜色等）
